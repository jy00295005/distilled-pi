# /pi-grill Reference

Use this workflow to run a concise PI-style research stress test from a required harness generated by `/title-to-harness`.

## Core Contract

Input:

```text
/pi-grill

Required:
<research-harness.md / survey-harness.md / benchmark-harness.md / dataset-harness.md / system-harness.md / position-harness.md>

Optional:
<abstract / proposal / draft / experiment plan / related work notes / PDF / slides>
```

The harness is mandatory. Do not run `/pi-grill` from an abstract, draft, PDF, or idea alone.

Final output:

- `pi-grill-report.md`

## Source of Truth

Treat the harness as the source of truth. Optional materials only sharpen questions, expose inconsistency, or add evidence.

Read these fields from the harness first:

- Original Title
- Paper Type
- Research Question
- Core Hypotheses
- Novelty Claim
- Minimum Viable Experiment
- Baselines
- Ablations
- Evaluation Metrics
- Falsification Conditions
- Known Risks
- Assumptions and Unknowns
- Questions for PI

If optional material conflicts with the harness, do not silently merge them. Emit:

```markdown
## Consistency Warning

The optional material seems to claim X, but the harness only supports Y.
This creates an overclaim risk.
```

Only override the harness if the user explicitly says it is outdated.

## Workflow

Follow this sequence:

```text
Step 0: Input Gate
Step 1: Material Parsing
Step 2: Meeting Diagnosis
Step 3: Round 1 - PI First Reaction
Step 4: User Answer Stress Test
Step 5: Round 2 - Targeted Follow-up if needed
Step 6: Final Repair Synthesis
Step 7: Generate pi-grill-report.md
```

This is not a reviewer checklist. Ask few questions. Make them count.

## Step 0: Input Gate

If no harness is provided, stop and say:

```text
I need the harness generated by /title-to-harness first.
You can also paste an abstract, draft, or PDF, but the harness is required.
```

Do not brainstorm from scratch.

Accept harnesses named:

- `research-harness.md`
- `survey-harness.md`
- `benchmark-harness.md`
- `dataset-harness.md`
- `system-harness.md`
- `position-harness.md`

Pasted harness Markdown is also acceptable.

## Step 1: Material Parsing

Extract the core fields. Keep the public diagnostic short.

```markdown
## Parsed Inputs

Harness Type:
...

Optional Materials:
- Abstract: yes/no
- Draft: yes/no
- PDF: yes/no
- Experiment plan: yes/no

Key Claim:
...

Key Risk:
...

Key Uncertainty:
...
```

If optional materials are thin, say:

```text
Optional materials were too thin to materially change the grill.
```

## Step 2: Meeting Diagnosis

Classify the stage:

- Harness stage
- Proposal stage
- Experiment design stage
- Preliminary results stage
- Draft paper stage
- Revision / rebuttal stage

Then identify the single main PI focus and main danger.

```markdown
## Meeting Diagnosis

Stage:
Proposal stage

PI likely focus:
Whether the proposed experiment can actually support the stated claim.

Main danger:
The current claim says "self-correction ability," but the experiment may only show "critique-conditioned answer editing."
```

Do not list every possible issue. Diagnose the most dangerous one.

## Step 3: Round 1 - PI First Reaction

Ask 2-3 questions only. Prefer 2 if enough.

Hard rules:

- Ask no more than 3 questions.
- Each question must be able to kill, reshape, or materially narrow the project.
- Do not ask generic checklist questions.
- Prioritize claim size, novelty, evidence, baselines, falsification, and type-specific contribution risk.

Format:

```markdown
## Round 1: PI First Reaction

### Q1. <fatal question>

Why this hurts:
...

What a strong answer needs:
...

### Q2. <fatal question>

Why this hurts:
...

What a strong answer needs:
...
```

Stop after Round 1 and wait for the user's answers unless the user explicitly asks for a final report using current assumptions.

## Type-Specific Fatal Questions

Use these as patterns, not as a checklist to dump.

### Research Harness

- Does the experiment prove the core claim, or only a weaker proxy?
- If the strongest baseline wins, what remains of the paper?
- Is the novelty a method, empirical finding, analysis, or just an engineering combination?
- What alternative explanation could explain all positive results?
- What result would make you admit the hypothesis is wrong?

### Survey Harness

- Why is the taxonomy an insight-generating framework rather than ordinary classification?
- Compared with recent surveys, what new judgment does this survey add?
- Will the comparison axes create insight, or only tables?
- What belief should readers change after reading it?
- How will this avoid becoming a literature pile?

### Benchmark Harness

- Does the benchmark measure the target capability, or a dataset artifact?
- If all models score similarly, is the benchmark still valuable?
- Why are existing benchmarks insufficient?
- How will you prove discriminative and diagnostic value?
- How vulnerable is it to benchmark gaming?

### Dataset Harness

- What phenomenon does this dataset capture that existing datasets miss?
- Why is this not ordinary data collection?
- How will data quality, bias, and annotation consistency be guaranteed?
- What new research questions does it enable?
- If gains come from scale rather than dataset design, what is the contribution?

### System Harness

- Is this solving a research problem or just implementing software?
- How does the evaluation prove workflow improvement?
- What is the core difference from existing tools?
- Why do users need it?
- Can the design insight become a paper contribution?

### Position / Perspective Harness

- Is the core position non-trivial enough?
- Which mainstream view does it challenge or refine?
- Is the evidence strong enough for the position?
- What is the strongest objection?
- What judgment should readers change?

## Step 4: User Answer Stress Test

After the user answers Round 1, evaluate each answer.

Verdicts:

- `Strong`
- `Okay`
- `Hand-wavy`
- `Dangerous`
- `Missing`

Format:

```markdown
## Answer Stress Test

### Q1 Verdict:
Hand-wavy

Why:
...

Stronger answer should:
- ...

Suggested repair:
...
```

Rules:

- Be direct, not mean.
- Do not over-explain.
- For every weak answer, provide a repair path.

## Step 5: Round 2 - Targeted Follow-up

Round 2 is optional.

Run it only if:

- A core answer is `Hand-wavy`, `Dangerous`, or `Missing`.
- One more question can materially improve the research plan.

Ask 1-2 questions only. Never more than 2.

Format:

```markdown
## Round 2: Targeted Follow-up

### Q1. <targeted question>

This is the key question because:
...

### Q2. <optional second question>

This is the key question because:
...
```

If Round 1 answers are strong enough, skip Round 2 and generate the report.

## Step 6: Final Repair Synthesis

Always converge to a repair plan, even if answers remain weak.

Include:

- PI's main concern
- Weakest claim
- Safer claim
- Evidence needed
- Experiments / analyses to add
- What to fix before next meeting
- Next 7-day plan
- Context for future AI work

Principle: first hit the weak point, then give the repair path.

## Style Rules

Sound like a real PI or senior advisor:

- Concise.
- Skeptical.
- High signal.
- No checklist dumping.
- No fake encouragement.
- No endless questions.

Acceptable short lines:

```text
This claim does not hold yet.
This experiment only proves the weaker version.
This is not a bad idea, but it is not the paper you think it is yet.
If the baseline wins, you need to know what is left.
This is worth doing, but the current claim is too big.
```

Always follow the hit with a repair:

```text
A safer claim is...
The next step is one diagnostic experiment...
This can shift from a method paper to an empirical analysis paper...
```
